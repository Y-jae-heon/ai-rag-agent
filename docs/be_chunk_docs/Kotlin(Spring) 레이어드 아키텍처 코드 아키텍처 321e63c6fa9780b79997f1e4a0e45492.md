## Controller

- 역할 : HTTP 요청 처리, 입력값 검증, 서비스 호출
- Swagger 문서화를 위해 `@Tags`, `@Operation` 어노테이션을 활용합니다.
- 검증 : `spring-boot-starter-validation`을 사용해 검증 어노테이션을 사용합니다.
- 경로 : 기본적으로 케밥 케이스를 사용하며, 자원은 복수형으로 표기합니다.
    - `/sensors/{id}/details`
- 리턴타입 : 일관성 있는 리턴타입을 위해 `ObjectResponse`, `ListResponse`를 사용합니다.

```kotlin
data class ObjectResponse<T>(
    val row: T
)

data class ListResponse<T>(
    val rows: List<T>,
    val count: Int
)
```

```kotlin
@RestController
@RequestMapping("sensors")
@SecurityRequirement(name = "bearerAuth")
class SensorController(
    private val sensorService: SensorService
) {

    @GetMapping("/{id}/details")
    @Operation(
        summary = "센서 상세 데이터 조회",
        description = "센서의 상세 데이터를 조회합니다. ISO String 사용"
    )
    fun getSensorDetail(
        @PathVariable("id") sensorId: Long,
        @ModelAttribute dto: FindSensorDetailReq
    ): ResponseEntity<ObjectResponse<FindSensorDetailRes>> {
        val sensor = sensorService.findDetailById(sensorId, dto)

        return ResponseEntity.ok(ObjectResponse(FindSensorDetailRes.from(sensor)))
    }
}
```

### 응답 코드

- 200 : 일반적인 데이터 리턴
- 201 : `POST` 요청으로 데이터가 생성되었을 때
- 204 : 삭제, 수정 등 작업 수행
- 400 : 잘못된 요청
- 401 : 인증 오류
- 403 : 인가 오류
- 404 : 자원 없음 (단일 자원일 때)
- 500 : 예상하지 못한 서버 오류

## Service

- 역할 : 비즈니스 로직 수행, 트랜잭션 관리
- 로직 : 서비스 레이어에서 로직은 도메인 로직의 오케스트레이션을 관리합니다. 핵심 도메인 로직은 엔티티에 작성합니다.
- 리턴타입 : 특별한 목적이 있는 메서드가 아니라면 기본적으로 DTO가 아닌 엔티티 형태로 리턴합니다.
    - 서비스 레이어의 재사용성을 높이기 위해 엔티티를 리턴하고, 최종 사용처에서 엔티티를 가공해서 사용합니다.

```kotlin
@Service
@Transactional(readOnly = true)
class OrderService(
    private val orderRepository: OrderRepository,
    private val productRepository: ProductRepository
) {

    @Transactional
    fun createOrder(dto: CreateOrderCommand): Order {
        // 1. 상품 조회
        val products = productRepository.findAllById(dto.productIds)

        // 2. 주문 엔티티 생성 (수량 검증 등 도메인 로직은 엔티티로 위임)
        val order = Order.create(
            memberId = dto.memberId,
            products = products,
            items = dto.items
        )

        // 3. 주문 저장
        orderRepository.save(order)

        // 4. 저장된 주문 리턴 (추후 재사용성을 위해)
        return order
    }
}
```

## Repository

- 역할 : 복잡한 DB 쿼리 및 데이터 접근 로직 캡슐화
- 구조 : JPA와 QueryDSL을 사용하기 때문에 Custom Repository 패턴을 사용합니다.

```kotlin
// 1. QueryDSL 전용 인터페이스 (명세 정의)
interface UserRepositoryCustom {
    fun searchUsers(cond: UserSearchCondition, pageable: Pageable): Page<User>
    fun findSummariesByTeam(teamId: Long): List<UserSummary>
}

// 2. QueryDSL 구현체
class UserRepositoryImpl(
    private val queryFactory: JPAQueryFactory
) : UserRepositoryCustom {

    override fun searchUsers(cond: UserSearchCondition, pageable: Pageable): Page<User> {
        // QueryDSL 쿼리 작성
    }

    override fun findSummariesByTeam(teamId: Long): List<UserSummary> {
        // ...
    }
}

// 3. 메인 Repository (JPA + Custom 통합)
interface UserRepository : JpaRepository<User, Long>, UserRepositoryCustom {
    fun findByEmail(email: String): User?
    fun existsByNickname(nickname: String): Boolean
}
```

- 동적쿼리, 생성쿼리의 파라미터는 DTO로 전달합니다. DTO 네이밍은 네이밍 컨벤션을 따릅니다.
- 검색 조건이 2개 이상인 경우 QueryDSL을 사용합니다.
- Repository 레이어에서는 트랜잭션을 선언하지 않습니다.
- 단일 조회 결과가 없는 경우 `null`을 리턴, 다중 조회 결과가 없는 경우 빈 리스트를 리턴합니다.
    - 조회 결과가 없을 때 예외가 되는 것은 비즈니스 로직이므로 서비스 계층에서 책임
    - 코틀린의 nullable 타입(`?`)을 활용해 null safety를 보장합니다.

### 페이지네이션

**오프셋**

- 페이지네이션은 `Pageable`, `Page` 객체를 사용합니다.
- 레포지토리 계층에서는 페이지네이션 결과를 `PaginationResult`로 변환해 리턴합니다.

```kotlin
data class PaginationResult<T>(
```

## Entity

- **역할** : 도메인 상태와 비즈니스 로직을 관리
- JPA 엔티티를 기본 엔티티로 사용합니다.
- 코틀린 JPA 엔티티는 **`open` class**로 선언합니다 (JPA proxy를 위해).
    - `kotlin-allopen` 플러그인으로 `@Entity`, `@MappedSuperclass`, `@Embeddable` 자동 open 처리를 권장합니다.
    - `kotlin-noarg` 플러그인으로 JPA가 요구하는 기본 생성자를 자동 생성합니다.
- `CoreEntity`를 상속해 감사 기능과 소프트 삭제를 기본 정책으로 사용합니다.

```kotlin
@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class CoreEntity {

    @CreationTimestamp
    @Column(name = "created_at", updatable = false, nullable = false)
    var createdAt: LocalDateTime? = null
        protected set

    @UpdateTimestamp
    @Column(name = "updated_at", nullable = false)
    var updatedAt: LocalDateTime? = null
        protected set

    @Column(name = "deleted_at")
    var deletedAt: LocalDateTime? = null
        protected set

    fun markDeleted() {
        this.deletedAt = LocalDateTime.now()
    }

    val isDeleted: Boolean
        get() = deletedAt != null
}
```

- 연관관계 필드는 최하단으로 몰아둡니다.
- `@Column`을 사용하여 컬럼 정보의 명시적 선언을 권장합니다.
    - `nullable`, `length` 등을 명시합니다.
- 도메인 무결성을 지키기 위해 프로퍼티의 `setter`를 `protected set` 또는 `private set`으로 제한합니다.
- ENUM 타입은 `EnumType.STRING`을 사용합니다.
- 식별자는 기본적으로 `GenerationType.IDENTITY`를 사용합니다.
- 코틀린의 `equals`/`hashCode` 오버라이딩 시 ID 기반으로 작성합니다 (data class 사용 금지).

### 연관관계

- 연관관계는 **단방향**을 기본으로 설정합니다.
    - 양방향 연관관계가 필요한 경우 연관관계를 설정할 수 있는 메서드를 사용합니다.

```kotlin
// Order (주인) 쪽에 편의 메서드 작성
fun addOrderItem(item: OrderItem) {
    this.orderItems.add(item)
    item.order = this  // 반대편도 함께 세팅
}
```

- 연관관계는 LAZY 로딩을 기본으로 합니다. `fetch = FetchType.LAZY`
- `@OneToMany`와 같은 컬렉션 필드는 `mutableListOf()`로 바로 초기화합니다.
- 외래키 제약조건을 사용하지 않습니다. `foreignKey = @ForeignKey(ConstraintMode.NO_CONSTRAINT)`

```kotlin
@Entity
@Table(name = "user")
class User(

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(nullable = false, updatable = false)
    val id: Long? = null,

    @Column(nullable = true)
    var azureId: String? = null,

    @Column(nullable = false, unique = true)
    var email: String,

    @Column(nullable = false)
    var name: String,

    @Column(nullable = false)
    var nickname: String,

    @Column(nullable = true)
    var krNickname: String? = null,

    @Column(nullable = false)
    var isActive: Boolean = false,

    var lastLoggedInAt: LocalDateTime? = null,

    // -- 연관관계 --

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(
        name = "department_id",
        foreignKey = ForeignKey(ConstraintMode.NO_CONSTRAINT)
    )
    var department: Department? = null,

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(
        name = "team_id",
        foreignKey = ForeignKey(ConstraintMode.NO_CONSTRAINT)
    )
    var team: Team? = null,

) : CoreEntity() {

    @BatchSize(size = 10)
    @OneToMany(mappedBy = "user", cascade = [CascadeType.ALL], orphanRemoval = true)
    val userRoles: MutableList<UserRole> = mutableListOf()
}
```

> **코틀린 엔티티 주의사항**
> 
> - `data class`를 엔티티로 사용하지 않습니다. (proxy 문제, equals/hashCode 이슈)
> - 변경 가능한 프로퍼티는 `var`, 불변 프로퍼티(ID 등)는 `val`로 선언합니다.
> - 생성자 프로퍼티로 선언하되, 컬렉션 연관관계는 body에 선언합니다.
