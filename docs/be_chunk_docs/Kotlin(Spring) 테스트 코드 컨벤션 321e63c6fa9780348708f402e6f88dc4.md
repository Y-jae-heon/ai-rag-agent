## 1. 기술 스택 및 의존성

### 1-1. 권장 의존성 (build.gradle.kts)

```kotlin
dependencies {
    // Kotest - Kotlin 전용 테스트 프레임워크 (권장)
    testImplementation("io.kotest:kotest-runner-junit5:5.8.0")
    testImplementation("io.kotest:kotest-assertions-core:5.8.0")
    testImplementation("io.kotest:kotest-property:5.8.0") // Property-based testing

    // Kotest Spring Extension
    testImplementation("io.kotest.extensions:kotest-extensions-spring:1.1.3")

    // MockK - Kotlin 전용 Mocking 라이브러리 (필수)
    testImplementation("io.mockk:mockk:1.13.9")
    testImplementation("com.ninja-squad:springmockk:4.0.2") // @MockkBean 지원

    // 코루틴 테스트
    testImplementation("org.jetbrains.kotlinx:kotlinx-coroutines-test:1.7.3")

    // AssertJ (Java 컨벤션과 혼용 시)
    testImplementation("org.assertj:assertj-core:3.24.2")

    // Spring Boot Test (기본 포함)
    testImplementation("org.springframework.boot:spring-boot-starter-test") {
        exclude(module = "mockito-core") // MockK 사용 시 Mockito 제거
    }
}

tasks.withType<Test> {
    useJUnitPlatform()
}
```

### 1-2. Java Spring과의 주요 차이점

| 항목 | Java Spring | Kotlin Spring |
| --- | --- | --- |
| Mocking | Mockito + `@Mock` | **MockK** + `@MockkBean` |
| 테스트 프레임워크 | JUnit5 | **Kotest** (권장) 또는 JUnit5 |
| Assertion | AssertJ | **Kotest Assertions** 또는 AssertJ |
| Fixture | static 팩토리 메서드 | **data class** + `copy()` 또는 Object 팩토리 |
| 테스트 명칭 | `@DisplayName` | **백틱(```)** 함수명 또는 `@DisplayName` |
| 코루틴 지원 | 없음 | `runTest`, `TestScope` |

## 2. 테스트 종류 및 범위

| 테스트 종류 | 어노테이션/도구 | 범위 | 속도 |
| --- | --- | --- | --- |
| 단위 테스트 | MockK + Kotest `DescribeSpec` | 단일 클래스 | 빠름 |
| 슬라이스 테스트 (Web) | `@WebMvcTest` + `@MockkBean` | Controller 레이어 | 보통 |
| 슬라이스 테스트 (DB) | `@DataJpaTest` | Repository 레이어 | 보통 |
| 통합 테스트 | `@SpringBootTest` + `@MockkBean` | 전체 컨텍스트 | 느림 |

**원칙**: 테스트 피라미드를 준수합니다. 단위 테스트를 최대한 많이 작성하고, 통합 테스트는 핵심 흐름에만 사용합니다.

```
        /\
       /  \        ← @SpringBootTest (적게)
      /----\
     /      \      ← @WebMvcTest, @DataJpaTest (적당히)
    /--------\
   /          \    ← Unit Test with MockK + Kotest (많이)
  /____________\
```

### 3-2. 테스트 함수명

Kotlin은 **백틱(```)** 으로 감싼 자연어 함수명을 지원합니다. 이를 적극 활용합니다.

```kotlin
// ✅ 백틱을 활용한 자연어 명칭 (권장)
@Test
fun `재고가 부족할 때 주문을 생성하면 OutOfStockException이 발생한다`() { }

// ✅ Kotest 스타일 - describe/it 블록 (가장 권장)
describe("주문 생성 시") {
    it("재고가 충분하면 주문이 생성된다") { }
    it("재고가 부족하면 OutOfStockException이 발생한다") { }
}

// ❌ 잘못된 예
@Test
fun test1() { }

@Test
fun testCreateOrder() { }
```

### 3-3. Kotest Spec 선택 기준

| Spec 종류 | 사용 시점 | 예시 |
| --- | --- | --- |
| `DescribeSpec` | 서비스/컨트롤러 계층 테스트, 그룹화가 필요할 때 | `describe("주문 생성") { it("성공한다") {} }` |
| `BehaviorSpec` | BDD 스타일 (given/when/then 블록) | `given("재고가 충분할 때") { when("주문하면") { then("성공") {} } }` |
| `FunSpec` | 단순한 함수형 테스트 | `test("주문이 생성된다") {}` |
| `StringSpec` | 가장 심플한 구조 | `"주문이 생성된다" {}` |

**팀 기본 Spec**: `DescribeSpec` 을 기본으로 사용합니다.

## 4. 테스트 스타일: Kotest vs JUnit5

### 4-1. Kotest DescribeSpec (권장)

```kotlin
@ExtendWith(SpringExtension::class)
class OrderServiceTest : DescribeSpec({

    val orderRepository = mockk<OrderRepository>()
    val orderService = OrderService(orderRepository)

    describe("createOrder") {

        context("재고가 충분할 때") {
            it("주문이 정상적으로 생성된다") {
                // given
                val request = OrderCreateRequest(memberId = 1L, productId = 1L, quantity = 2)
                val savedOrder = OrderFixture.createOrder()
                every { orderRepository.save(any()) } returns savedOrder

                // when
                val result = orderService.createOrder(request)

                // then
                result.status shouldBe OrderStatus.PENDING
                result.quantity shouldBe 2
            }
        }

        context("재고가 부족할 때") {
            it("OutOfStockException이 발생한다") {
                // given
                val request = OrderCreateRequest(memberId = 1L, productId = 1L, quantity = 100)
                every { orderRepository.findProductStock(any()) } returns 5

                // when & then
                shouldThrow<OutOfStockException> {
                    orderService.createOrder(request)
                }
            }
        }
    }
})
```

### 4-2. JUnit5 스타일 (레거시 호환 또는 팀 선택)

```kotlin
@ExtendWith(MockKExtension::class)
class OrderServiceTest {

    @InjectMockKs
    private lateinit var orderService: OrderService

    @MockK
    private lateinit var orderRepository: OrderRepository

    @Test

        // given
        val request = OrderCreateRequest(memberId = 1L, productId = 1L, quantity = 2)
        every { orderRepository.save(any()) } returns OrderFixture.createOrder()

        // when
        val result = orderService.createOrder(request)

        // then
        assertThat(result.status).isEqualTo(OrderStatus.PENDING)
    }
}
```

## 5. 테스트 구조 (Given-When-Then)

모든 테스트는 **Given-When-Then** 패턴으로 작성합니다.

### 5-1. Kotest DescribeSpec에서의 GWT

```kotlin
describe("cancelOrder") {
    context("대기 중인 주문을 취소할 때") {
        it("주문 상태가 CANCELLED로 변경된다") {
            // given
            val order = OrderFixture.createPendingOrder()
            every { orderRepository.findById(1L) } returns order

            // when
            orderService.cancelOrder(1L)

            // then
            order.status shouldBe OrderStatus.CANCELLED
        }
    }
}
```

### 5-2. BehaviorSpec에서의 GWT (명시적 블록 사용)

```kotlin
class OrderServiceBehaviorTest : BehaviorSpec({
    given("재고가 충분한 상품이 있을 때") {
        val product = ProductFixture.createProductWithStock(10)

        `when`("수량 2개 주문하면") {
            val result = orderService.createOrder(OrderCreateRequest(quantity = 2))

            then("주문이 생성된다") {
                result.status shouldBe OrderStatus.PENDING
            }
        }
    }
})
```

**규칙**:

- `// given`, `// when`, `// then` 주석 또는 Kotest의 `context`/`it` 블록 구조로 섹션을 명확히 합니다.
- `when` 섹션(또는 `it` 블록 내 행위)은 단일 동작 하나만 포함합니다.
- Kotest의 `context`는 조건, `it`은 기대 결과를 서술합니다.

## 6. 단위 테스트 (Unit Test)

### 6-1. MockK를 활용한 서비스 단위 테스트

```kotlin
class OrderServiceTest : DescribeSpec({

    // MockK: mockk<T>()로 생성
    val orderRepository = mockk<OrderRepository>()
    val memberRepository = mockk<MemberRepository>()
    val productRepository = mockk<ProductRepository>()

    // 생성자 주입으로 SUT 생성 (lateinit 불필요)
    val orderService = OrderService(orderRepository, memberRepository, productRepository)

    // 각 테스트 전 초기화
    beforeEach {
        clearAllMocks()
    }

    describe("createOrder") {
        context("유효한 요청이 들어오면") {
            it("주문이 정상적으로 생성된다") {
                // given
                val member = MemberFixture.create()
                val product = ProductFixture.createWithStock(10)
                val request = OrderCreateRequest(member.id, product.id, 2)

                every { memberRepository.findById(member.id) } returns member
                every { productRepository.findById(product.id) } returns product
                every { orderRepository.save(any()) } answers { firstArg() }

                // when
                val result = orderService.createOrder(request)

                // then
                result.status shouldBe OrderStatus.PENDING
                result.quantity shouldBe 2
                verify(exactly = 1) { orderRepository.save(any()) }
            }
        }
    }
})
```

### 6-2. 도메인 객체 단위 테스트

외부 의존성이 없는 도메인 객체는 MockK 없이 순수하게 테스트합니다.

```kotlin
class OrderTest : DescribeSpec({

    describe("Order 생성 시") {
        context("수량이 0 이하이면") {
            it("IllegalArgumentException이 발생한다") {
                shouldThrow<IllegalArgumentException> {
                    Order(quantity = 0)
                }.message shouldBe "주문 수량은 1개 이상이어야 합니다."
            }
        }
    }

    describe("isCancellable") {
        context("주문 상태가 PENDING이면") {
            it("취소 가능하다") {
                val order = Order(status = OrderStatus.PENDING, quantity = 1)
                order.isCancellable() shouldBe true
            }
        }

        context("주문 상태가 SHIPPED이면") {
            it("취소 불가능하다") {
                val order = Order(status = OrderStatus.SHIPPED, quantity = 1)
                order.isCancellable() shouldBe false
            }
        }
    }
})
```

## 8. 슬라이스 테스트

### 8-1. Controller 테스트 (`@WebMvcTest`)

```kotlin
@WebMvcTest(OrderController::class)
class OrderControllerTest(
    @Autowired private val mockMvc: MockMvc,
    @Autowired private val objectMapper: ObjectMapper,
) : DescribeSpec({

    val orderService = mockk<OrderService>()

    describe("POST /api/orders") {
        context("유효한 요청이면") {
            it("201 Created를 반환한다") {
                // given
                val request = OrderCreateRequest(memberId = 1L, productId = 1L, quantity = 2)
                val response = OrderResponse(id = 1L, status = OrderStatus.PENDING, quantity = 2)

                every { orderService.createOrder(any()) } returns response

                // when & then
                mockMvc.perform(
                    post("/api/orders")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(objectMapper.writeValueAsString(request))
                )
                    .andExpect(status().isCreated)
                    .andExpect(jsonPath("$.id").value(1L))
                    .andExpect(jsonPath("$.status").value("PENDING"))
                    .andDo(print())
            }
        }

        context("수량이 없으면") {
            it("400 Bad Request를 반환한다") {
                // given
                val request = mapOf("memberId" to 1L, "productId" to 1L) // quantity 누락

                // when & then
                mockMvc.perform(
                    post("/api/orders")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(objectMapper.writeValueAsString(request))
                )
                    .andExpect(status().isBadRequest)
            }
        }
    }
}) {
    @MockkBean
    lateinit var orderService: OrderService
}
```

## 9. Fixture 및 테스트 데이터 관리

### 9-1. data class + copy()를 활용한 Fixture

Kotlin의 `data class`와 `copy()`를 활용하면 Java의 Builder 패턴보다 간결한 Fixture를 만들 수 있습니다.

```kotlin
// src/test/kotlin/com/example/fixture/MemberFixture.kt
object MemberFixture {

    // 기본 Fixture - 가장 일반적인 케이스
    fun create(
        id: Long = 1L,
        email: String = "test@example.com",
        name: String = "테스트유저",
        status: MemberStatus = MemberStatus.ACTIVE,
    ): Member = Member(
        id = id,
        email = email,
        name = name,
        status = status,
    )

    fun createInactive(): Member = create(status = MemberStatus.INACTIVE)

    fun createWithEmail(email: String): Member = create(email = email)
}
```

```kotlin
// 사용 예 - named parameter로 원하는 필드만 오버라이드
val member = MemberFixture.create()
val inactiveMember = MemberFixture.create(status = MemberStatus.INACTIVE)
val specificMember = MemberFixture.create(id = 99L, email = "specific@test.com")
```

### 9-2. data class copy()를 활용한 변형

```kotlin
object OrderFixture {

    fun createOrder(
        id: Long = 1L,
        member: Member = MemberFixture.create(),
        quantity: Int = 1,
        status: OrderStatus = OrderStatus.PENDING,
    ): Order = Order(id = id, member = member, quantity = quantity, status = status)

    fun createPendingOrder() = createOrder(status = OrderStatus.PENDING)
    fun createShippedOrder() = createOrder(status = OrderStatus.SHIPPED)
    fun createCancelledOrder() = createOrder(status = OrderStatus.CANCELLED)
}
```

**규칙**:

- Fixture object는 `src/test/kotlin` 하위 `fixture` 패키지에 위치합니다.
- 모든 파라미터에 기본값을 제공하여, 테스트에서 필요한 부분만 오버라이드합니다.
- `object` 키워드로 싱글턴으로 선언합니다 (Java의 static 메서드와 동일한 용도).

### 9-3. @ParameterizedTest 활용

```kotlin
// Kotest - withData로 파라미터 테스트
class OrderCancellableTest : DescribeSpec({
    describe("isCancellable") {
        withData(
            nameFn = { "status=${it.first} 이면 cancellable=${it.second}" },
            OrderStatus.PENDING to true,
            OrderStatus.CONFIRMED to true,
            OrderStatus.SHIPPED to false,
            OrderStatus.DELIVERED to false,
            OrderStatus.CANCELLED to false,

    "CANCELLED, false"

fun `주문 상태에 따라 취소 가능 여부가 결정된다`(status: OrderStatus, expected: Boolean) {
    val order = OrderFixture.createOrder(status = status)
    assertThat(order.isCancellable()).isEqualTo(expected)
}

## 10. Assertion 작성 원칙

### 10-1. Kotest Assertions 사용 (권장)

Kotest를 사용하는 경우, `shouldBe` 계열의 Kotest Assertions를 사용합니다.

```kotlin
// ✅ Kotest Assertions (infix 스타일 - 가독성 우수)
result.status shouldBe OrderStatus.PENDING
result.id shouldNotBe null
orders shouldHaveSize 3
name shouldStartWith "테스트"
message shouldContain "재고 부족"

// ✅ 예외 검증
shouldThrow<OutOfStockException> {
    orderService.createOrder(invalidRequest)
}.message shouldBe "재고가 부족합니다."

shouldNotThrow<Exception> {
    orderService.createOrder(validRequest)
}

// ❌ assertTrue/assertEquals 단독 사용 금지 (Kotest 사용 시)
assertTrue(result.isCancellable())
assertEquals(OrderStatus.PENDING, result.status)

## 11. Mocking 전략 (MockK)

Kotlin에서는 **Mockito 대신 MockK**를 사용합니다. Kotlin의 `final` 클래스, `data class`, 코루틴 등을 자연스럽게 지원합니다.

### 11-1. 기본 MockK 사용법

```kotlin
// ✅ mockk<T>() - 기본 Mock 생성
val orderRepository = mockk<OrderRepository>()

// ✅ every { } - stubbing (BDDMockito의 given() 역할)
every { orderRepository.findById(1L) } returns OrderFixture.createOrder()
every { orderRepository.save(any()) } answers { firstArg() } // 첫 번째 인자 그대로 반환
every { orderRepository.findAll() } returns emptyList()

// ✅ verify { } - 호출 검증 (BDDMockito의 then().should() 역할)
verify(exactly = 1) { orderRepository.save(any()) }
verify(exactly = 0) { emailSender.send(any()) } // 호출 안됨 검증

// ✅ confirmVerified - 검증되지 않은 호출이 없음을 확인
confirmVerified(orderRepository)
```

### 11-2. Unit 반환 함수 Mocking

```kotlin
// ✅ Unit 반환 함수 (void 메서드) stubbing
every { emailSender.send(any()) } just Runs

// ✅ Unit 반환 함수 예외 발생
every { emailSender.send(any()) } throws RuntimeException("메일 발송 실패")
```

### 11-3. relaxed Mock

```kotlin
// ✅ relaxed = true - 모든 함수가 기본값 반환 (stub 없이 사용 가능)
val emailSender = mockk<EmailSender>(relaxed = true) // Unit 반환 함수는 just Runs 불필요

// ✅ relaxedUnitFun = true - Unit 반환 함수만 relaxed
val service = mockk<OrderService>(relaxUnitFun = true)
```

### 11-4. Spy (부분 Mocking)

```kotlin
// ✅ spyk<T>() - 실제 구현 유지하면서 일부만 Mocking
val calculator = spyk(OrderPriceCalculator())
every { calculator.calculateDiscount(any()) } returns 1000

// 검증
verify { calculator.calculateDiscount(any()) }
```

### 11-5. Captor (인자 캡처)

```kotlin
// ✅ capture()로 인자 캡처 후 검증
val orderSlot = slot<Order>()
every { orderRepository.save(capture(orderSlot)) } answers { firstArg() }

orderService.createOrder(request)

val capturedOrder = orderSlot.captured
capturedOrder.status shouldBe OrderStatus.PENDING
capturedOrder.quantity shouldBe 2
```

### 11-6. MockK vs Mockito 비교

```kotlin
// Mockito (Java 스타일) - Kotlin에서 사용 시 문제 발생 가능
given(orderRepository.findById(anyLong())).willReturn(Optional.of(order)) // Optional 불필요
`when`(orderRepository.findById(1L)).thenReturn(order) // when이 Kotlin 예약어와 충돌

// ✅ MockK (Kotlin 전용) - 더 자연스러운 문법
every { orderRepository.findById(1L) } returns order // Optional 없이 nullable 타입 처리
```

## 12. 코루틴 테스트

Kotlin에서 suspend 함수를 포함하는 코드를 테스트할 때의 방법입니다.

### 12-1. runTest 사용

```kotlin
class OrderServiceTest : DescribeSpec({

    val orderRepository = mockk<OrderRepository>()
    val orderService = OrderService(orderRepository)

    describe("createOrderAsync") {
        context("유효한 요청이면") {
            it("주문이 비동기로 생성된다") {
                // runTest: 코루틴 테스트 전용 스코프
                // 가상 시간을 사용하여 delay() 등을 즉시 실행
                runTest {
                    // given
                    coEvery { orderRepository.saveAsync(any()) } returns OrderFixture.createOrder()

                    // when
                    val result = orderService.createOrderAsync(OrderCreateRequest(1L, 1L, 2))

                    // then
                    result.status shouldBe OrderStatus.PENDING
                }
            }
        }
    }
})
```

### 12-2. MockK 코루틴 지원

```kotlin
// ✅ suspend 함수 stubbing: coEvery { }
coEvery { orderRepository.findByIdAsync(1L) } returns OrderFixture.createOrder()

// ✅ suspend 함수 검증: coVerify { }
coVerify(exactly = 1) { orderRepository.saveAsync(any()) }

// ✅ suspend Unit 함수
coEvery { emailSender.sendAsync(any()) } just Runs
```

### 12-3. 딜레이 및 타임아웃 테스트

```kotlin
it("3초 안에 주문이 완료된다") {
    runTest(timeout = 3.seconds) {
        // given
        coEvery { orderRepository.saveAsync(any()) } coAnswers {
            delay(100) // 100ms 딜레이 시뮬레이션
            firstArg()
        }

        // when & then
        val result = orderService.createOrderAsync(request)
        result.status shouldBe OrderStatus.PENDING
    }
}
```

## 14. 금지 패턴 (Anti-Patterns)

### 14-1. 테스트에서 절대 금지

```kotlin
// ❌ Thread.sleep 사용 금지 - 코루틴 테스트에서는 runTest + delay 사용
Thread.sleep(1000)

// ❌ println 사용 금지 - 로거 또는 andDo(print()) 사용
println("결과: $response")

// ❌ !! (non-null assertion) 남용 금지 - NPE 원인 불명확
val result = orderService.createOrder(request)!!.id!! // 금지

// ✅ shouldNotBeNull() 사용
val result = orderService.createOrder(request)
result.id.shouldNotBeNull()

// ❌ 비어있는 테스트 금지
it("주문이 생성된다") {
    // TODO
}

// ❌ 테스트 내 조건 분기 금지
it("상태에 따라 결과가 다르다") {
    if (condition) { // 금지 - 각각 별도 it 블록으로 분리
        result shouldBe A
    } else {
        result shouldBe B
    }
}
```

### 14-2. Kotlin + MockK 관련 금지 패턴

```kotlin
// ❌ Mockito를 Kotlin 코드에서 사용 금지 (final 클래스 이슈)
@Mock
lateinit var orderRepository: OrderRepository // MockK 사용

// ❌ any() 대신 타입이 명확할 때 구체적인 값 사용
every { repository.findById(any()) } returns order // ❌ 의도 불명확
every { repository.findById(1L) } returns order    // ✅ 명확한 의도

// ❌ MockK에서 Mockito 스타일 혼용 금지
`when`(repository.findById(1L)).thenReturn(order) // Mockito 스타일 금지
every { repository.findById(1L) } returns order   // ✅ MockK 스타일

// ❌ @MockkBean 없이 @MockBean 사용 금지 (final 클래스 Kotlin에서 문제 발생)
@MockBean
lateinit var orderService: OrderService // ❌

@MockkBean
lateinit var orderService: OrderService // ✅
```

### 14-3. Kotlin 특성 관련 금지 패턴

```kotlin
// ❌ data class equals/hashCode를 무시한 검증
assertThat(result).isSameAs(expected) // 참조 비교 - 금지
result shouldBe expected              // ✅ data class equals 비교

// ❌ 불필요한 nullable 처리
val result: Order? = orderService.createOrder(request)
assertThat(result!!.status).isEqualTo(OrderStatus.PENDING) // !! 금지

// ✅ 명시적 null 검증 후 처리
result.shouldNotBeNull()
result.status shouldBe OrderStatus.PENDING
```

## 15. 패키지 구조

```
src/
├── main/
│   └── kotlin/com/example/
│       └── domain/
│           └── order/
│               ├── Order.kt
│               ├── OrderController.kt
│               ├── OrderService.kt

└── test/
    └── kotlin/com/example/
        ├── domain/
        │   └── order/
        │       ├── OrderTest.kt                   # 도메인 단위 테스트
        │       ├── OrderControllerTest.kt         # 컨트롤러 슬라이스 테스트
        │       ├── OrderServiceTest.kt            # 서비스 단위 테스트
        │       └── OrderRepositoryTest.kt         # 레포지토리 슬라이스 테스트
        ├── integration/
        │   └── OrderIntegrationTest.kt            # 통합 테스트
        └── fixture/
            ├── MemberFixture.kt                   # 회원 픽스처 (object)
            ├── ProductFixture.kt                  # 상품 픽스처 (object)
            └── OrderFixture.kt                    # 주문 픽스처 (object)
```

**규칙**:

- 테스트 패키지 구조는 프로덕션 코드 패키지 구조를 그대로 따릅니다.
- `fixture` 패키지는 `test` 루트에 공통으로 위치합니다.
- 통합 테스트는 별도 `integration` 패키지로 분리합니다.
- Fixture 클래스는 `object` 키워드를 사용합니다.
