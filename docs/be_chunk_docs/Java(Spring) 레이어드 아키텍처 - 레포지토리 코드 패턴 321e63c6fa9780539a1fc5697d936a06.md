## Rule

### Repository

- 역할 : 복잡한 DB 쿼리 및 데이터 접근 로직 캡슐화
- 구조 : JPA와 QueryDSL을 쓰기 때문에 Custom Repository 패턴을 사용한다.
    
    ```java
    // 1. QueryDSL 전용 인터페이스 (명세 정의)
    public interface UserRepositoryCustom {
        Page<User> searchUsers(UserSearchCondition cond, Pageable pageable);
        List<UserSummary> findSummariesByTeam(Long teamId);
    }
    
    // 2. QueryDSL 구현체
    public class UserRepositoryImpl implements UserRepositoryCustom {
        private final JPAQueryFactory queryFactory;
    
        @Override
        public Page<User> searchUsers(UserSearchCondition cond, Pageable pageable) {
            // QueryDSL 쿼리 작성
        }
    }
    
    // 3. 메인 Repository (JPA + Custom 통합)
    public interface UserRepository extends JpaRepository<User, Long>,
                                            UserRepositoryCustom {
        // Spring Data JPA 메서드 선언
        Optional<User> findByEmail(String email);
        boolean existsByNickname(String nickname);
    }
    
    ```
    
- 동적쿼리 파라미터는 DTO 로 전달한다. DTO네이밍은 네이밍 컨벤션을 따른다.
- 만약 검색 조건이 2개 이상 넘어가는 경우 QueryDSL을 사용한다.
- Repository 레이어에서는 트랜잭션을 선언하지 않는다.
- 단일 조회 결과가 없는경우 `NULL` 을 리턴, 다중 조회 결과가 없는 경우 빈 배열 을 리턴한다.
    - 조회 결과가 없을때 예외가 되는것은 비즈니스 로직이므로 서비스 계층에서 책임을 지는것이 명확
    - 만약 조회만 하고 싶은 경우 (동일 이름 확인등) 예외가 터지기 문제 발생.

**생성 Create 메서드**

- 생성 메서드의 파라미터는 엔티티를 사용합니다.
- 생성 메서드의 리턴값으로 엔티티를 리턴합니다.

**조회 Read 메서드**

- 조회 메서드 파라미터는 DTO를 사용합니다.
- 페이지네이션이 필요하다면 페이지네이션 정보를 같이 전달합니다.

**수정 Update 메서드**

- 수정 메서드의 파라미터는 엔티티를 사용합니다.

**삭제 Delete 메서드**

- 삭제 메서드는 엔티티의 ID를 통해 SoftDelete 합니다.

### 페이지네이션

**오프셋**

- 페이지네이션은 `Pageable` , `Page` 객체를 사용한다.

**카운트 쿼리**

- 최적화가 필요하지 않다면 기본적으로 `Page` 를 사용해 카운트 쿼리를 분리하지 않습니다.

커서 방식

- 커서 방식 페이지네이션이 필요하다면 `Window` 객체를 사용합니다.
