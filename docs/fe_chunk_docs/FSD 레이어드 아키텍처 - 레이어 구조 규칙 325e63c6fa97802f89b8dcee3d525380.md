# Title

FSD 레이어 구조 규칙

# Rule

프로젝트는 다음 레이어 구조를 사용한다.

```
src
 ├ app
 ├ pages
 ├ widgets
 ├ features
 ├ entities
 └ shared
```

레이어 정의

### app

애플리케이션 초기화 구성 요소, app에 ui는 레이아웃 등 전역 레이아웃 템플릿이 들어갈 수 있다.

예

```
app/providers
app/routing
app/styles
app/ui
```

### pages

라우트 단위 페이지

예

```
pages/user
pages/dashboard
pages/station
```

### widgets

페이지 블록 UI

예

```
widgets/dashboard-analytics
widgets/user-management-panel
```

### features

사용자 행동 기능

예

```
features/user/create
features/user/update
features/auth/login
```

### entities

도메인 모델

예

```
entities/user
entities/station
entities/auth
```

### shared

전역 공통 모듈

예

```
shared/ui
shared/api
shared/lib
shared/types
```

# Rationale

레이어 구조는 다음 목적을 가진다.

- 코드 의존성 방향 통제
- 도메인 중심 구조 유지
- UI / 비즈니스 로직 분리

또한 레이어 간 의존성 규칙을 통해 다음 문제를 방지한다.

- 순환 의존성
- 도메인 로직 오염
- 공통 모듈 남용

# Exception

특정 도메인에서 복합 기능이 필요한 경우 `processes` 레이어를 추가할 수 있다.

processes 레이어는 **여러 feature를 묶어 하나의 비즈니스 흐름(workflow)을 구성할 때 사용하는 선택적 레이어**이다. 즉, **단일 기능(feature)이 아니라 “여러 기능의 orchestration”을 담당한다.**

예시 비즈니스 흐름 → 회원가입

예

```
processes/checkout
processes/onboarding
processes/sign-up
```

# Override

레이어 추가 또는 제거는 Architecture Decision Record(ADR)를 통해 결정한다.
