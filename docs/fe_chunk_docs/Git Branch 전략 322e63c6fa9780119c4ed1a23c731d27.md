## Title

Git Branch Strategy Convention

---

## Rule

프로젝트는 **main / develop 기반 브랜치 전략**을 사용한다.

브랜치는 두 가지 유형으로 나뉜다.

유지 브랜치

작업 브랜치

---

### 유지 브랜치

main

운영 배포 중인 코드가 위치하는 브랜치이다.

develop

개발 통합 브랜치이다.

기능 개발이 완료된 feature 브랜치가 병합된다.

---

### 작업 브랜치

feature

기능 개발을 위한 브랜치이다.

feature 브랜치는 develop 브랜치에서 생성한다.

예

```
feature/coupon/list
```

---

release

운영 배포 전 QA 작업을 수행하는 브랜치이다.

develop에서 생성한다.

release 브랜치에서는 배포 대상 기능만 유지한다.

---

hotfix

운영 환경에서 발생한 긴급 버그 수정 브랜치이다.

main 브랜치에서 생성한다.

---

## Rationale

브랜치 전략은 다음 문제를 해결한다.

- 기능 개발 충돌 방지
- QA 단계 분리
- 안정적인 배포 관리

---

## Exception

작은 프로젝트에서는 develop 브랜치를 생략할 수 있다.

---

## Override

프로젝트에 따라 trunk based development 전략을 사용할 수 있다.
