## Title

프론트엔드 파일 네이밍 컨벤션 (FSD 기반)

---

## Rule

### 1. React 컴포넌트

컴포넌트 파일은 **PascalCase**를 사용한다.

예

```
LoginForm.tsx
UserAvatar.tsx
DashboardPanel.tsx
```

주로 다음 위치에서 사용된다.

```
ui
widgets
pages
```

---

### 2. Hook 파일

Hook 파일은 **use-{kebab-case}.ts 형식**을 사용한다.

Hook에는 다음이 포함된다.

- React Hook
- TanStack Query Hook
- 비즈니스 로직 Hook

예

```
use-auth.ts
use-login-query.ts
use-login-mutation.ts
use-user-profile.ts
```

위 파일은 주로 다음 위치에 존재한다.

```
features/{slice}/{sementic}/hooks
entities/{slice}/hooks
shared/hooks
```

---

### 3. API 파일

API 파일은 **kebab-case**를 사용한다.

예

```
login-api.ts
create-user.ts
fetch-order-list.ts
```

주로 다음 위치에 존재한다.

```
features/{slice}/api
entities/{slice}/api
shared/api
```

---

## 4. Model 관련 파일

Model 관련 파일은 **model 폴더 내부에 위치한다.**

model은 다음 데이터를 정의할 수 있다.

```
schema
dto
types
state
domain model
validation
```

---

### Entities Model 예시

entities는 **도메인 객체 모델을 정의한다.**

예

```
entities/user/model
 ├ user.schema.ts
 ├ user.dto.ts
 ├ user.types.ts
```

---

### Features Model 예시

features는 **행동 단위 데이터 구조를 정의한다.**

예

```
features/auth/login/model
 ├ login.schema.ts
 ├ login.dto.ts
```

또는

```
features/user/create/model
 ├ create-user.schema.ts
 ├ create-user.dto.ts
```

---

### 5. Utility 파일

Utility 파일은 **kebab-case**를 사용한다.

예

```
format-date.ts
parse-query.ts
build-request-url.ts
```

주로 다음 위치에서 사용된다.

```
shared/lib
features/{slice}/lib
entities/{slice}/lib
```

---

### 6. Test 파일

테스트 파일은 다음 형식을 사용한다.

```
{kebab-case}.spec.ts
```

예

```
login.spec.ts
user-api.spec.ts
use-login-mutation.spec.ts
```

---

### 7. Asset 파일

정적 리소스 파일은 **kebab-case**를 사용한다.

예

```
loading-spinner.svg
user-avatar.png
arrow-right.svg
```

위 파일은 다음 위치에 존재한다.

```
shared/assets
```

---

## Rationale

파일 네이밍 규칙의 목표는 다음과 같다.

- 파일명만으로 역할 식별 가능
- IDE 검색 효율 증가
- slice 기반 구조와 semantic 유지
- model 폴더 내부 의미 구분

특히 model 폴더의 경우

```
schema
dto
types
state
```

를 **폴더로 세분화하지 않고 파일 네이밍으로 구분**한다.

이 방식은

- 구조 단순화
- RAG 검색 효율 증가
- 개발자 인지부하 감소

효과가 있다.

---

## Exception

다음 경우 예외를 허용한다.

- 외부 라이브러리 규칙을 따라야 하는 파일
- 프레임워크 자동 생성 파일
- 레거시 코드

---

## Override

프로젝트 기술 스택 변경 시 파일 네이밍 규칙 override를 허용한다.

override 시

- 변경 사유
- 영향 범위
- 마이그레이션 계획

을 문서화해야 한다.
