# Title

프론트엔드 FSD 아키텍처 개요

# Rule

프론트엔드 아키텍처는 **Feature-Sliced Design(FSD)** 구조를 기반으로 설계된다.

FSD는 프론트엔드 코드를 **도메인 중심으로 구조화하고 의존성 방향을 명확하게 통제하기 위한 아키텍처 패턴**이다. 이 구조를 통해 프로젝트 규모가 커지더라도 코드의 탐색성과 유지보수성을 유지할 수 있도록 한다.

본 프로젝트는 다음과 같은 레이어 구조를 사용한다.

```bash
app
pages
widgets
features
entities
shared
```

각 레이어는 서로 다른 역할을 가지며 다음과 같이 정의된다.

**app 레이어는 애플리케이션의 전역 초기화 영역이다.**

이 레이어에는 전역 스타일 설정, React Provider 구성, 라우팅 설정, i18n 초기화와 같은 애플리케이션 시작 시 필요한 설정이 위치한다.

**pages 레이어는 라우트 단위의 페이지를 구성하는 영역이다.**

페이지는 URL 경로와 직접 연결되는 UI이며, 실제 화면을 구성하기 위해 entities, widgets, features UI를 조합하는 역할을 한다.

**widgets 레이어는 페이지 단위의 UI 블록을 구성하는 영역이다.**

widgets는 여러 feature 또는 entity UI를 조합하여 만들어지는 비교적 큰 UI 단위이며, 주로 대시보드 패널이나 관리 화면 패널과 같은 페이지 블록을 표현한다.

**features 레이어는 사용자의 행동이나 인터랙션을 구현하는 기능 단위 영역이다.**

이 레이어에는 로그인, 회원 생성, 데이터 수정, 삭제와 같은 사용자 행동 중심 기능이 포함된다. Feature는 특정 도메인에 속하며, UI와 비즈니스 로직을 함께 포함하는 기능 단위로 구성된다.

**entities 레이어는 도메인 모델을 표현하는 영역이다.**

이 레이어는 서버의 도메인 모델과 유사한 개념을 가지며, 사용자(User), 상품(Product), 충전소(Station)와 같은 핵심 도메인 데이터를 표현한다. 또한 해당 도메인의 조회 로직이나 도메인 기반 UI도 포함될 수 있다.

**shared 레이어는 프로젝트 전역에서 공통적으로 사용하는 모듈을 포함하는 영역이다.**

여기에는 범용 UI 컴포넌트, 공통 유틸리티 함수, HTTP 클라이언트, 타입 정의, 환경 설정과 같은 공통 모듈이 포함된다.

또한 FSD 아키텍처에서는 **레이어 간 의존성 방향을 엄격하게 관리한다.**

레이어는 다음 방향으로만 의존할 수 있다.

```bash
app → pages → widgets → features → entities → shared
```

즉, 상위 레이어는 하위 레이어를 참조할 수 있지만 하위 레이어는 상위 레이어를 참조할 수 없다. 이 규칙은 프로젝트 구조가 커졌을 때 발생할 수 있는 순환 의존성과 도메인 오염을 방지하기 위한 것이다.

본 프로젝트에서는 UI 구조를 설계할 때 **Atomic Design 패턴을 폴더 구조로 사용하지 않는다.**

Atomic Design은 UI 컴포넌트를 설계할 때 참고하는 개념적 모델로만 사용하며, 실제 폴더 구조에서는 다음과 같은 위치 기준을 따른다.

범용 UI 컴포넌트는 `shared/ui`에 위치한다.

도메인에 종속적인 UI는 `entities/{domain}/ui`에 위치한다.

사용자 행동과 관련된 UI는 `features/{feature}/ui`에 위치한다.

페이지 블록 단위 UI는 `widgets/{widget}/ui`에 위치한다.

이러한 규칙을 통해 UI 컴포넌트의 위치를 명확하게 결정할 수 있으며, Atomic Design 구조로 인해 발생할 수 있는 UI 계층 혼란을 방지할 수 있다.

# Rationale

FSD 아키텍처를 사용하는 이유는 다음과 같다.

첫째, 프로젝트 규모가 커질 때 도메인 중심 구조를 유지하기 위해서이다.

둘째, UI 코드와 비즈니스 로직을 명확하게 분리하기 위해서이다.

셋째, 레이어 기반 의존성 규칙을 통해 코드 결합도를 낮추기 위해서이다.

넷째, 여러 개발자가 동시에 작업하더라도 코드 위치와 책임 범위를 쉽게 이해할 수 있도록 하기 위해서이다.

또한 UI 구조에서 Atomic Design을 폴더 구조로 사용하지 않는 이유는 다음과 같다.

Atomic Design을 폴더 구조로 적용하면 UI 분류 기준이 증가하게 되고, FSD의 slice 구조와 중복되는 구조가 발생할 수 있다. 이러한 구조는 컴포넌트의 위치를 판단하기 어렵게 만들고 관리 비용을 증가시킬 수 있다.

따라서 Atomic Design은 컴포넌트 설계 개념으로만 사용하고, 실제 폴더 구조는 FSD 기준에 따라 구성한다.

# Exception

소규모 프로젝트에서는 FSD 레이어 구조를 단순화할 수 있다.

예를 들어 다음과 같은 구조를 사용할 수 있다.

```bash
features
shared
```

이 경우에도 도메인 중심 구조와 공통 모듈 분리 원칙은 유지하는 것이 권장된다.

# Override

특정 프로젝트에서 구조 변경이 필요할 경우 Architecture Review 또는 Architecture Decision Record(ADR)를 통해 구조 변경 여부를 결정한다.
