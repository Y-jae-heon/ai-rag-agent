## Title

프론트엔드 폴더 네이밍 컨벤션 (FSD Layered Architecture)

---

## Rule

### 1. 레이어 폴더

프로젝트의 최상위 구조는 **FSD Layer 기준으로 구성한다.**

허용되는 레이어는 다음과 같다.

```
app
pages
widgets
features
entities
shared
```

예

```
src/app
src/pages
src/widgets
src/features
src/entities
src/shared
```

---

## 2. Slice 폴더 (도메인)

`features`, `entities`, `widgets` 레이어 내부의 slice 폴더는

**kebab-case를 사용한다.**

단, **features 레이어는 도메인 행동(action) 기준으로 구성한다.**

---

### Entities (도메인 객체)

entities는 **도메인 객체 기준**으로 구성한다.

예

```
entities
 ├ user
 ├ order
 ├ product
```

---

### Features (도메인 행동)

features는 **도메인 행동 기준으로 구성한다.**

예

```
features
 ├ user
 │   ├ create
 │   ├ update
 │   └ duplicate-check
 │
 ├ auth
 │   ├ login
 │   └ logout
```

---

### Widgets

widgets는 **페이지 블록 UI 기준**으로 구성한다.

예

```
widgets
 ├ dashboard-panel
 ├ user-profile-card
 └ payment-summary
```

---

### 3. Feature 내부 구조

feature slice 내부는 **역할(segment) 기준으로 구성한다.**

허용되는 주요 segment

```
ui
api
hooks
model
lib
store
```

예

```
features/login
 ├ ui
 ├ api
 ├ hooks
 ├ model
 └ lib
```

---

### 4. Entity 구조

entity는 **도메인 데이터와 데이터 관련 로직을 담당한다.**

예

```
entities/user
 ├ ui
 ├ api
 ├ model
 └ lib
```

---

### 5. Model 폴더

`model` 폴더는 **도메인 데이터 구조와 데이터 규칙을 정의하는 영역이다.**

model 폴더에는 다음과 같은 파일이 포함될 수 있다.

예

```
schema
dto
types
state
domain rules
validation
```

예시

```
model
 ├ user.schema.ts
 ├ user.dto.ts
 ├ user.types.ts
 └ auth-state.ts
```

model은 **폴더를 세분화하지 않는다.**

schema / dto / types 등은 **파일 네이밍 규칙으로 구분한다.**

파일 네이밍 규칙은

**"프론트엔드 파일 네이밍 컨벤션" 문서를 참조한다.**

---

### 6. Shared 구조

shared는 **전역에서 재사용되는 공통 모듈이다.**

예

```
shared/ui
shared/api
shared/lib
shared/config
shared/types
shared/assets
shared/hooks
shared/store
```

---

### 7. Pages 구조

pages는 **라우트 단위 화면을 의미한다.**

페이지 폴더는 **kebab-case**를 사용한다.

예

```
pages/login
pages/user-profile
pages/payment-history
```

---

## Rationale

FSD 아키텍처는 **도메인 중심 구조**를 기반으로 한다.

폴더 네이밍 규칙을 다음 기준으로 유지한다.

- 레이어 기반 구조 유지
- 도메인 중심 slice 구성
- 역할(segment) 기반 내부 구조
- 파일명보다 **폴더 구조로 의미 파악 가능**

model 폴더를 별도 세분화하지 않는 이유는 다음과 같다.

- 구조 복잡도 증가 방지
- slice 내부 semantic 유지
- 폴더 깊이 증가 방지

대신 **파일 네이밍 규칙을 통해 semantic을 구분한다.**

---

## Exception

다음 상황에서는 예외를 허용한다.

- 프레임워크 구조가 요구되는 경우
- 외부 라이브러리 구조를 따라야 하는 경우
- 레거시 코드와 호환해야 하는 경우

예

```
app/router
app/providers
```

---

## Override

프로젝트 아키텍처 변경 시 폴더 구조 override를 허용한다.

override 적용 시

- 변경 사유
- 적용 범위
- 영향 범위

를 문서화해야 한다.
