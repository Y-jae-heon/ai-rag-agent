# Title

Git Pull Request Convention

---

# Rule

Pull Request(PR)는 코드 변경 사항을 협업자에게 설명하고 코드 리뷰를 진행하기 위한 문서이다.

PR은 단순히 코드 리뷰를 위한 절차가 아니라 다음 목적을 가진다.

- 협업자 간 변경 사항 공유
- Jira 기반 작업 이력 관리
- 코드 품질 유지
- 협업 커뮤니케이션 기록

PR은 프로젝트 상황과 협업 방식에 따라 유연하게 운영한다.

---

# PR 규칙

PR은 프로젝트 일정과 협업 상황을 고려하여 운영한다.

프로젝트 일정상 **빠른 머지가 필요한 경우**

- 리뷰 절차를 간소화할 수 있다.
- 휴면 이슈 방지를 위해 최소 확인 후 머지를 진행한다.

리뷰 진행 여유가 있는 경우

- 리뷰어를 지정하여 리뷰를 요청한다.
- 리뷰 완료 후 머지를 진행한다.

리뷰어 선정 규칙

- 매달 말일 기준으로 **3명 이상의 리뷰 그룹을 지정한다.**
- 그룹 단위로 PR 리뷰를 진행한다.

---

# PR 단위

PR 생성 기준은 프로젝트 상황에 따라 유동적으로 운영한다.

기본 원칙

- 모든 feature 단위 작업에 대해 PR을 생성하는 것을 권장한다.

협업 프로젝트

- 여러 명이 작업하는 경우 프로젝트 내부 규칙을 따른다.
- 일반적으로 feature 단위 PR 생성이 기본 원칙이다.
- 프로젝트 일정이나 상황에 따라 유연하게 운영할 수 있다.

개인 프로젝트

- 혼자 프로젝트를 진행하는 경우에도 PR을 생성할 수 있다.
- 다른 프로젝트 인원에게 코드 리뷰를 요청할 수 있다.

권장 방식

- 혼자 작업하는 개발자들도 PR 리뷰를 주고받는 습관을 유지하는 것을 권장한다.

---

# PR 작성 및 머지 규칙

PR은 기능 개발 단위(feature)가 완료되었을 때 작성한다.

기본 흐름

feature 작업 완료

→ develop 브랜치 대상 PR 생성

→ 코드 리뷰 진행

→ PR 머지

PR 머지는 다음 상황에 따라 유연하게 진행할 수 있다.

- 리뷰 완료 후 머지
- 일정상 빠른 반영이 필요한 경우 최소 확인 후 머지

---

# PR 설명 작성 규칙

PR 설명은 **이해 관계자가 변경 내용을 쉽게 이해할 수 있도록 작성한다.**

PR 설명에는 다음 항목을 포함한다.

개요

PR 내용

PR 리뷰 포인트

PR 추가 설명

참고 이미지

# Rationale

Pull Request는 협업 과정에서 코드 변경을 설명하는 핵심 문서이다.

PR 규칙을 유지하면 다음 효과가 있다.

- 코드 리뷰 품질 향상
- 변경 이력 추적 용이
- 협업 커뮤니케이션 기록 유지
- 코드 품질 유지

또한 PR 기반 협업은 개발자 간 지식 공유를 촉진한다.

---

# Exception

다음 상황에서는 PR 규칙을 완화할 수 있다.

긴급 hotfix

서비스 장애 대응

운영 장애 긴급 수정

이 경우 최소 확인 후 빠르게 머지할 수 있다.

---

# Override

프로젝트 상황에 따라 다음 규칙은 변경될 수 있다.

- PR 리뷰 프로세스
- 리뷰어 지정 방식
- PR 생성 기준

팀의 협업 방식에 맞게 PR 정책을 조정할 수 있다.
